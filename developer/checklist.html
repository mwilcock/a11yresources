  • This checklist cannot be used to verify conformance with WCAG 2.0.

  • This checklist should not be referenced in policies or in policy adoption.

  • This checklist contains interpretation of WCAG guidelines and success criteria and our own recommended techniques for satisfying those success criteria.

1.1.1 - Non-Text Content

Tags:

The purpose of this Success Criterion is to make information conveyed by non-text content accessible using a text alternative. Text alternatives are a primary way of making information accessible because they can be rendered through any sensory modality (for example, visual, auditory or tactile) to match the needs of the user.

Why do this

This Success Criterion benefits many users, but primarily those that have difficulty interpreting visual content. Assistive technologies can read text alternatives aloud, present it visually in a manner a user can better understand, or convert it to braille. Users that have a difficulty interpreting the meaning of visual content also benefit from this Success Criterion, as the description acts as a provided interpretation of the visual content. Lastly, providing a text alternative for non-text content means users can search for the visual content through its text alternative (such as through as a search bar feature).

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Not app Yes Yes Not app

What to do

To successfully implement this Success Criterion, endeavour to:

  • Add text alternatives for all images (HTML alt=”” attribute).
  • Add a text alternative for videos and audio, by providing an accurate description of the topic of the video/audio. (Note: There is more in-depth Success Criterion for Audio and Video content, which should also be adhered to).
  • Ensure all controls such as buttons or search bars are labelled explicitly with text, such as adding ‘Submit’ onto buttons, ‘Search’ by all search bars, etc.

Tips

Remember that providing alternative text for visual content should substitute it, not add on to it. This means alternative text should only provide an interpretation of the content that is visually displayed. Of course, this isn’t always fully achievable, but it is still worth providing an accurate text alternative as possible, just focus on what is being displayed.

Exceptions

Non-Text content such as decorative information or logos are not required to have a text-alternative attributed to them since they are for decoration and do not convey information to the user, other than being aesthetic. This applies to decorative elements such as spacers, used for formatting page content.

Implementing Non-Text Content

Attribute alt-text to all non-text elements within the HTML mark-up of the application, and provide an accurate description of the content within this alt attribute. (alt=”description of image”). Images/Non-Text content that is exempt from the Success Criterion should still be provided with an alt attribute, however, should be given a null value. (alt=””). This ensures that Assistive Technologies skip the content with a null alt attribute completely. This prevents screen readers from reading out the filename or location of the image which could confuse screen reader users.

Screen Reader
  • Move screen reader focus to each image.
  • Make sure that the screen reader announces a meaningful text alternative for the image and that no content inside the image is missing from the spoken alternative.
  • Make sure that purely-decorative and meaningless images are not focusable.

1.2.1 - Audio Only and Video Only (Pre-Recorded)

Tags:

The purpose of this Success Criterion is to make information conveyed by pre-recorded audio-only and pre-recorded video-only (video with no audio) content available to all users. Alternatives for time-based media that are text based make information accessible because text can be rendered through any sensory modality (for example, visual, auditory or tactile) to match the needs of the user.

Why do this

This Success Criterion benefits many users, but primarily those that have difficulty interpreting visual content. Assistive technologies can read text alternatives aloud, present it visually in a manner a user can better understand, or convert it to braille. Users that have difficulty understanding the meaning of video content are provided with an alternative that allows them to interpret the video content, this also applies for users that have a difficulty interpreting audio content, as an alternative is provided for them to interpret.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Not app Yes Yes Not app

What to do

To successfully implement this Success Criterion, endeavour to:

  • Provide a text alternative for all audio-only content present within the application, that accurately conveys the information portrayed within the audio-only content.
  • Provide a text alternative for all video-only content present within the application that accurately conveys the information portrayed in the video-only content. Alternatively, a media alternative such as an audio description of the video-only content could be provided for users that benefit from alternate representations of text.

Tips

Alternatives for audio-only and video-only content should be a substitution for the video/audio content it is for, not an addition or augment to it. This is easily done by providing a transcript for audio-only content and providing an audio track that presents equivalent information for video-only content.

Exceptions

Text equivalents are not required for audio that is provided as an equivalent for video-only content. For example, an audio description of an animation with no audio does not need a text alternative, as the audio track is an alternative for the animation.

Implementing Audio Only and Video Only (Pre-Recorded)

Providing text-alternatives for audio-only content could be done by linking to a text-transcript of the audio. Video-only content can be provided with either a text alternative, such as a description of the video-only content, which can be placed next to the video or linked to externally, or by providing an audio track that conveys equivalent information within the video-only content, which can be placed by the video-only content or linked to externally.

1.2.3 - Audio Description or Media Alternative (Pre-Recorded)

Tags:

The purpose of this Success Criterion is to provide blind or visually impaired users access to the visual information within synchronised media (video with audio). This can be done one of two ways, by either providing an audio description of the video content which augments the original audio track (think of it as an overlay) and is a description of the visual content or by providing both the visual and auditory information in a text format. An alternative to synchronised media provides a description of everything that occurs within the media content.

Why do this

This Success Criterion benefits users who may have difficulty interpreting video or any other synchronised media, including those that have difficulty interpreting or understanding moving images.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
partial Yes Yes Not app

What to do

To successfully implement this Success Criterion, endeavour to:

  • Either provide an audio description that describes visual cues within synchronised media, or
  • Provide a text-alternative to the synchronised media that describes everything that occurs within the video and audio content of the synchronised media.

Tips

The alternative for time-based media should read like a screenplay or book. The description of the video content is not constrained to just the pauses in dialogue; full descriptions should be provided for all visual information, including visual context, actions and expressions of actors, and any other visual material. All non-speech sounds (laughter, off-screen voices, etc.) should be described, and transcripts of all dialogue should be included. Therefore, the alternative for time-based media can provide a much more complete representation of the synchronised media content than just providing an audio description.

Exceptions

This Success Criterion does not apply to media that is an alternative for text content and is clearly labelled as such.

Implementing Audio Description or Media Alternative (Pre-Recorded)

If providing an audio description, this may be done by providing a link to a version of the synchronised media with an audio description. A text-alternative can also be linked to an external file/page that contains the text-alternative.

1.2.4 - Captions (Live)

Tags:

This Success Criterion enables users that are deaf or hard of hearing to watch real-time synchronised media presentations (video with audio) and interpret the audio content through the captions provided.

Why do this

This Success Criterion benefits users that are deaf or hard of hearing, as it allows them to interpret live audio within media content through captions provided.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Not app Not app Yes Not app

What to do

To successfully implement this Success Criterion, endeavour to:

  • Provide accurate captions for all live synchronised media that is present in the application.

Tips

Accurate captions not only include dialogue but identify who is speaking and include non-speech information conveyed through sound, including meaningful sound effects.

Exceptions

If live audio is, for example, a two-way multimedia call between two or more individuals, then it is up to the content providers (the call host) to provide captions and not the application.

Implementing Captions (Live)

Providing captions for live content is in-depth and costly to implement, however, can be done. A common way to do this is to send audio signals from the live content through a caption encoder, which then sends these to a closed-caption provider.

1.3.1 - Info and Relationships

Tags:

The purpose of this success criterion ensures that visually grouped content is defined programmatically, so blind or visually impaired users can still interpret groups of content without having to interpret visual cues such as paragraphs being separated by blank lines, or headings being in large bold text. Defining such features programmatically using correct semantic definitions such as using <:heading> tags for headings, <:paragraph> tags for paragraphs of text, etc. means that AT interprets these as such and thus allows AT users to gain an understanding of the visual structure and relationships between content without requiring visual context.

Why do this

This Success Criterion benefits users with a range of disabilities and impairments that view web pages with different styling than set by the application. For example, some users may set up their browsers to show only the HTML of web pages (without CCS styling), or remove images from the page. All information is still available to these users regardless of how they view it.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Yes Yes Yes Yes

What to do

To successfully implement this criterion, endeavour to:

  • Use correct semantic tags for elements such as headings, bold text, paragraphs etc.
  • Mark required fields in forms with text (as well as colour if this is a feature)
  • Programmatically define labels for elements such as radio buttons, checkboxes, drop-down menus and buttons.

Tips

No tips (yet) for this Success Criterion.

Exceptions

There are no exceptions for this Success Criterion.

Implementing Info and Relationships

Tags:

The purpose of this Success Criterion is to make information conveyed by pre-recorded audio-only and pre-recorded video-only (video with no audio) content available to all users. Alternatives for time-based media that are text based make information accessible because text can be rendered through any sensory modality (for example, visual, auditory or tactile) to match the needs of the user.

Why do this

  • People with physical disabilities often need more time to react, to type and to complete activities. People with low vision need more time to locate things on the screen and to read. People who are blind and using screen readers may need more time to understand screen layouts, to find information and to operate controls. People who have cognitive or language limitations need more time to read and to understand. People who are deaf and communicate in sign language may need more time to read information printed in text (which may be a second language for some).
  • In circumstances where a sign-language interpreter may be relating audio content to a user who is deaf, control over time limits is also important.
  • People with reading disabilities, cognitive limitations, and learning disabilities who may need more time to read or comprehend information can have additional time to read the information by pausing the content.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Yes Yes Yes Yes

What to do

There are multiple situations where time limits are a factor for users. The situations below outline the different situations and how it can be adjusted to meet this criterion:

  • If there are session time limits
    • Providing a checkbox on the first page of a multipart form that allows users to ask for a longer time limit or that there is no time limit at all.
    • Provide a way for users to disable the time limits such as through the use of a button.
  • If a time limit is controlled by scripting on the page.
    • Provide users with a way to disable the time limit.
    • Provide users with the functionality to set time limit to 10 times the default time limit.
    • Provide a notification that the time limit is going to expire and then allowing users to further extend this time limit.
  • If there is a time limit for reading content.
    • Allow content to be paused and restarted from the same point in which it was paused.
    • Provide users with the ability to turn off any time limits.
    • Provide a mechanism to allow users to stop scrolling content.

Tips

  • In times where time limits are essential but can be changed to suit a user’s requirements such as in a timed test, the ability for a third party to control time limits such as providing extra time in the testing scenario is beneficial.

Exceptions

  • If the time limit of an action or event is dependent on real-time such as in a live auction, then no time alternative is possible.
  • If the time limit provided is essential to the activity or event and by extending the time it would invalidate the activity an exception is made.
  • If a time limit is longer than 20 hours (Longer than a full waking day), then an exception is made as this is deemed to be enough time to fulfil all tasks.

2.2.2 - Pause, Stop, Hide

Tags:

This success criterion is to ensure that users that are easily distracted by certain content within web pages can avoid this content should they wish to. Content that is included within this success criterion is content that conveys a sense of motion and is auto-updating. This includes but is not limited to motion pictures, animations, real-time games or scrolling news banners. Content that moves or is auto-updated can cause issues with users who struggle to read the stationary text quickly as well as users who find it difficult to track moving objects. Content that moves can also be an extreme distraction for some users such as those with attention deficit disorders.

Why do this

  • Providing content that stops blinking after five seconds or providing a mechanism for users to stop blinking content allows people with certain disabilities to interact with the Web page.
  • One use of content that blinks is to draw the visitor's attention to that content. Although this is an effective technique for all users with vision, it can be a problem for some users if it persists. For certain groups, including people with low literacy, reading and intellectual disabilities, and people with attention deficit disorders, content that blinks may make it difficult or even impossible to interact with the rest of the Web page.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Yes Not app Not app Not app

What to do

  • Allow users to pause and restart content from the same point it was paused.
  • Create content that blinks for less than 5 seconds.
  • Use technologies that allow any blinking content to be turned off via browser functionality.
  • Set animations to stop blinking or playing after several cycles (Within 5 seconds)
  • Provide controls for content on the page that allows the content that is moving, blinking or auto-updating to be paused or stopped.
  • Provide functionality to reload the page without blinking content.

Tips

  • For functionality to be considered a mechanism for the user to pause, stop or hide content the user must not become locked in the focus of an element to do so; the page must be operable without the need to remain focused on an element.
  • The five-second exception does not apply to auto-updating content as it would be redundant to have content stop updating after a period of five seconds.

Exceptions

Animation as part of a loading or preloading page may be essential to ensure that users of the site do not confuse loading with a frozen site.

2.4.1 - Bypass Block

Tags:

The purpose of this success criterion is to allow users who navigate web pages sequentially through content a way to navigate directly to the main content of the specific page. Web pages often have content that appears on pages or screens repeatedly. An example of such repeated content and the most common use is navigational mechanisms. Other examples include headings graphics, adverts and contents. Minor repeated actions or content is not considered to be a block. This mechanism is not required on all blocks of content unless increases a user’s navigational productivity.

Why do this

  • When this Success Criterion is not satisfied, it may be difficult for people with some disabilities to reach the main content of a Web page quickly and easily.
  • Screen reader users who visit several pages on the same site can avoid having to hear all heading graphics and dozens of navigation links on every page before the main content is spoken.
  • People who use only the keyboard or a keyboard interface can reach content with fewer keystrokes. Otherwise, they might have to make dozens of keystrokes before reaching a link in the main content area. This can take a long time and may cause severe physical pain for some users.
  • People who use screen magnifiers do not have to search through the same headings or other blocks of information to find where the content begins each time they enter a new page.
  • People with cognitive limitations as well as people who use screen readers may benefit when links are grouped into lists

  • Add a link at the start of a block of repeated content to go to the end of the block.
  • Add a list of links at the top of the page to each area of the page.
  • Provide ARIA landmarks to identify regions of the page.
  • Provide headings at the beginning of each section of content.

2.4.3 - Focus Order

Tags:

The purpose of this success criterion is to ensure that when users navigate web page content sequentially, they are presented with content in an order which is consistent with the meaning of the content and can be operated from the keyboard. This reduces confusion by letting users form a consistent mental model of the content to refer to. The sequential navigational order is determined by the web technology used to define the content. I.e. HTML defines the sequential navigation via the tabbing order. To outline:

  • Elements that receive focus must do so in an order which maintains the meaning and operable functions whereby the order of navigation affects the meaning or operable function.
  • In cases where this criterion applies there may be multiple orders that preserve the meaning and operability.
  • Where there is more than one order to preserve the meaning and operability only a single order should be provided.

Why do this

  • These techniques benefit keyboard users who navigate documents sequentially and expect the focus order to be consistent with the sequential reading order.
  • People with mobility impairments who must rely on keyboard access for operating a page benefit from a logical, usable focus order.
  • People with disabilities that make reading difficult can become disoriented when tabbing takes focus someplace unexpected. They benefit from a logical focus order.
  • People with visual impairments can become disoriented when tabbing takes focus someplace unexpected or when they cannot easily find the content surrounding an interactive element.
  • Only a small portion of the page may be visible to an individual using a screen magnifier at a high level of magnification. Such a user may interpret a field in the wrong context if the focus order is not logical.

    • Using a tab index property
    • Ensuring the DOM matches the visually displayed order

Tips

  • Ensure that focus order is logical for all users and does not appear to leap about the page in a seemingly random order.

3.2.2 - On Input

Tags:

The purpose of this success criterion is to ensure that entering data or selecting a form control has a predictable effect on the page. Changing the setting of any user interface components is changing some aspect of the control that will remain when the user is no longer interacting with the control. i.e. selecting a checkbox or entering data into a text box is changing the setting but activating a button or link is not. Changes of context without notification can confuse users who cannot easily perceive the change or are easily distracted by changes. Changes of context are only appropriate if the user is notified of such a change.

Why do this

This Success Criterion helps users with disabilities by making interactive content more predictable. Unexpected changes of context can be so disorienting for users with visual disabilities or cognitive limitations that they are unable to use the content. Individuals who are unable to detect changes of context are less likely to become disoriented while navigating a site. For example Individuals who are blind or have low vision may have difficulty knowing when a visual context change has occurred, such as a new window popping up. In this case, warning users of context changes in advance minimize confusion when the user discovers that the back button no longer behaves as expected. Some individuals with low vision, with reading and intellectual disabilities, and others who have difficulty interpreting visual cues may benefit from additional cues in order to detect changes of context.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Yes Yes Not app Yes

What to do

To successfully implement this Success Criterion, endeavour to:

  • Provide a submit button to initiate a change of context.
  • Using a button with a select element to perform an action.

Tips

  • Changes in content are not always a change of context.
  • Changes in context include changing one of the following:
    • User agent
    • Viewport
    • Focus
    • Content that changes the meaning of the web page.

Exceptions

Changes in content that are not a change in context examples include:

  • Expanding outlines
  • Dynamic menus
  • Tab controls

3.3.2 - Labels or Instructions

Tags:

The purpose of this criterion is to ensure that authors place instructions or labels that identify the controls in a form so that that user can understand what data is expected. Instructions or labels may also specify data formats in a form so that users know what input data is expected. This is not intended to clutter the page with information but important cues and instructions that will benefit people with disabilities and impairments should be used.

Why do this

When label elements are associated with input elements the label is spoken by screen readers when the field receives focus and users with impaired motor control are helped by a larger clickable area for the control, since clicking on the label or the control will activate the control. Field labels located in close proximity to the associated field assist users of screen magnifiers because the field and label are more likely to visible within the magnified area of the page. Providing examples of expected data formats help users with cognitive, language and learning disabilities to enter information correctly. Clearly identifying required fields prevents a keyboard only user from submitting an incomplete form and having to navigate the redisplayed form to find the uncompleted field and provide the missing information.

Cognitive
Visual
Auditory
Mobility

To successfully implement this Success Criterion, endeavour to:

  • Provide descriptive and meaningful labels.
  • Maximise predictability of labels via visual positioning.
  • Use label elements to associate text labels with form controls.
  • Provide descriptions for groups of form controls.
  • Use ARIA to provide further expansion for assistive technologies.

Tips

When labels are provided for input objects, the input object's relationship to the label (or to redundant text serving as the label) must be programmatically determinable or available in text.

3.3.3 - Error Suggestion

Tags:

The purpose of this criterion is to ensure that users receive appropriate suggestions for correcting input errors. This input error could be information that has been omitted or it could also be information which is provided incorrectly. The content author should provide a description of the error that has occurred and provide a suggestion to fix this; it is also possible that the error suggestion can come from the user agent should the information be provided programmatically.

Why do this

Providing information about how to correct input errors allows users who have learning disabilities to fill in a form successfully. Users who are blind or have impaired vision understand more easily the nature of the input error and how to correct it. People with motion impairment can reduce the number of times they need to change an input value.

4.1.1 - Parsing

Tags:

Assistive technologies use the markup or code to accurately interpret and parse content. If the content cannot be parsed into a data structure, then different assistive technologies may interpret content differently or be completely unable to parse the content. Some Assistive technologies use Repair techniques on badly written code. But as Repair Techniques differ, authors of content cannot assure that the Assistive technology will properly interpret the content.

Why do this

Ensuring that Web pages have complete start and end tags and are nested per specification helps ensure that assistive technologies can parse the content accurately and without crashing.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Yes Yes Yes Yes

What to do

To successfully implement this Success Criterion, endeavour to:

  • Ensure completed tags (i.e. Brackets, angle brackets and closing tags)
  • Nest elements appropriately
  • No duplicate attributes for each element.
  • IDs should be unique

Tips

Start and end tags that are missing a critical character in their formation, such as a closing angle bracket or a mismatched attribute value quotation mark are not complete.

4.1.2 - Name, Role, Value

Tags:

The purpose of this criterion is to ensure that Assistive technologies can gather information about, activate (Or set) and keep up to date on the status of user interface controls within the content. Custom controls that are defined to provide a different function or have a different role must have additional measures to ensure that the controls provide important information to assistive technologies and can be controlled by assistive technologies. This is achieved automatically when correctly implementing default HTML user interface controls; custom controls must replicate this functionality.

Why do this

Providing role, state, and value information on all user interface components enables compatibility with assistive technology, such as screen readers, screen magnifiers, and speech recognition software, used by people with disabilities.

This Success Criterion is applicable to the following disabilities:

Cognitive
Visual
Auditory
Mobility
Yes Yes Yes Yes

What to do

To successfully implement this Success Criterion, endeavour to:

  • Ensure form controls are given appropriate names and labels
  • Implement standard user interface components per specification.
  • Ensure all custom controls have programmatically determinable name, roles and values.

Tips

Whereby a visual name cannot be defined use of ARIA-Label to provide an invisible label that can be read by Assistive technologies. Use of ARIA-Labeledby to provide a name for all user interface controls.

No exceptions to this criterion.
